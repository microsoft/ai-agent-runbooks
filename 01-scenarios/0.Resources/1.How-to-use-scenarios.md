# How to Use Scenarios

> **Purpose:** This guide explains how the `01-scenarios/` folder is organized,
> how to navigate scenario-specific runbooks, and how to contribute new scenarios.  
> **Audience:** CSA, Vendor Engineer, Vendor Architect, CFTL  
> **Related folders:** `01-runbooks/` (phase-based delivery guides) · `03-patterns/`(reusable technical patterns)

---

## Scenario Folder Structure Overview

The `01-scenarios/` folder contains **scenario-specific runbooks** for each supported
CAF agent delivery scenario. Each scenario folder is self-contained and follows a
standardized structure so that any CSA or vendor team can pick it up and deliver
consistently — regardless of region or prior experience with the scenario.

```
01-scenarios/
├── HR-Onboarding-Agent/   ← 🔨 In progress
│   ├── Overview.md
│   ├── Architecture.md
│   ├── Runbook.md
│   └── Sample-Prompts.md
├── Scenario02(TBD)/       ← 🚧 Coming Soon
├── Scenario03(TBD)/       ← 🚧 Coming Soon
├── Scenario04(TBD)/       ← 🚧 Coming Soon
└── Scenario05(TBD)/       ← 🚧 Coming Soon
```

> 💡 **Tip:** Always start with the scenario's `Overview.md` to confirm the scenario
> matches your customer's use case and CAF eligibility criteria before proceeding
> with delivery.

## Scenario Folder Structure

Each scenario folder follows a standardized 4-file structure to ensure consistency,
reusability, and clarity across all CAF agent deliveries.

Below is the standard structure, using **HR Onboarding Agent** as the reference example:

| File | Description |
|------|-------------|
| `Overview.md` | High-level summary of the scenario: supported CAF category (e.g., Employee Self-Service), business value, customer eligibility criteria, supported technical patterns, and expected delivery outcomes. Start here before opening a runbook. |
| `Architecture.md` | Logical architecture diagram and detailed explanation of key components, data flows, integration points, and authentication model. Reuse the pre-generated architecture diagrams stored here. |
| `Runbook.md` | Scenario-specific delivery procedures for **Phase 1 → Phase 2 → Phase 3**, customized from the standard phase runbooks in `01-runbooks/` to reflect the specific requirements, integrations, and configurations of this scenario. |
| `Sample-Prompts.md` | Example conversations with the agent covering primary use cases, edge cases, and error handling scenarios. Use these as reference inputs for UAT test case design and training material preparation. |

## How to Navigate Scenarios During a CAF Engagement

Follow this decision flow when starting a new CAF engagement:

**Step 1.** Confirm scenario category with customer  
↓
**Step 2.** Open `01-scenarios/` → locate matching scenario folder  
↓
**Step 3.** Read `Overview.md` → confirm CAF eligibility and pattern fit  
↓
**Step 4.** Review `Architecture.md` → validate against customer environment  
↓
**Step 5.** Follow `Runbook.md` → Phase 1 → Phase 2 → Phase 3  
↓
**Step 6.** Use `Sample-Prompts.md` → UAT test case preparation & training  

> ⚠️ **If no matching scenario folder exists** for the customer's use case:
> - Check `02-patterns/` for reusable technical patterns that can be combined
> - Review the [CAF Supported Scenarios](https://microsoft.sharepoint.com/sites/ABSCSU/SitePages/Agent-CAF.aspx#technical-and-business-scenarios)
>   to confirm the use case is in CAF scope
> - If the scenario is new but in-scope, follow the **How to Add a New Scenario**
>   section below to contribute a new scenario folder

## How to Add a New Scenario

When a new agent scenario has been successfully delivered through CAF and is ready
to be codified as a reusable runbook, follow these steps to contribute it:

### Step 1 — Confirm the scenario is in CAF scope

- Verify the scenario maps to a supported CAF pattern
  (see [Agent Scenarios deck](https://microsoft.sharepoint.com/sites/ABSCSU/Shared%20Documents/FY26/CAF/Cloud%20Accelerate%20Factory%20(For%20Agents)%20-%20Agent%20Scenarios.pptx))
- Confirm the scenario has been **successfully delivered at least once** in a
  real CAF engagement before creating a runbook

### Step 2 — Create the scenario folder

```
02-scenarios/
└── Your-New-Scenario-Name/    ← use kebab-case naming
    ├── Overview.md
    ├── Architecture.md
    ├── Runbook.md
    └── Sample-Prompts.md
```

> Use the **HR-Onboarding-Agent/** folder as your reference template.

### Step 3 — Fill in each file

| File | Minimum Required Content |
|---|---|
| `Overview.md` | Scenario name, CAF category, business value, eligibility criteria, supported patterns, expected outcomes |
| `Architecture.md` | Logical architecture diagram, component list, data flow description, integration map |
| `Runbook.md` | Phase 1–3 customized steps based on `01-runbooks/` standard guides, with scenario-specific configurations highlighted |
| `Sample-Prompts.md` | ≥ 5 primary use case examples + ≥ 3 edge case / error handling examples |

### Step 4 — Update this file

- Add the new scenario to the **Currently Supported Scenarios** table above
- Update status from `🚧 Coming Soon` → '🔨 In progress' → `✅ Available` once the runbook is complete

### Step 5 — Submit for review

- Refer to [CONTRIBUTING.md](../../CONTRIBUTING.md) for detailed contribution
  guidelines, review process, and approval requirements before merging

## Scenario Naming Convention

Use **kebab-case** for all scenario folder and file names to ensure consistency
across the repository:

| ✅ Correct | ❌ Incorrect |
|---|---|
| `HR-Onboarding-Agent/` | `HR Onboarding Agent/` |
| `Contact-Center-Agent/` | `ContactCenterAgent/` |
| `Sample-Prompts.md` | `SamplePrompts.md` |
