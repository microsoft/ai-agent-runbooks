# Sample Prompts and Query Catalog

> The Dynamics 365 Monitoring Agent ships with a library of **pre-curated queries designed
> for D365 Finance & Operations** environments. These queries represent the most common
> monitoring needs identified by support engineers and system administrators.
>
> This library is continuously growing as new platforms are integrated and new telemetry
> signals become available. We are actively collaborating with engineers across different
> Dynamics 365 and Power Platform teams to expand and refine the query catalog, making
> the agent progressively more capable with each iteration.
>
> Because the entire Dynamics 365 and Power Platform ecosystem exports telemetry to
> **Application Insights** following the same principles, this agent is inherently
> **multi-purpose**. The same framework, the same topics, the same flows. The only
> difference is which queries are configured in the Dataverse table.
>
> New queries can be added at any time by inserting rows into the **MCSA Agent Queries**
> table, making the agent progressively more capable without any code changes.

---

> **⚙️ First-Time Setup**
>
> Before running any query, you must configure the agent's connection to your Application Insights
> instance. Open the agent in Copilot Studio, start a test conversation, and type:
> *"Open the agent configuration center"*. Update the **Application Insights Configuration**
> with your tenant ID, resource group name, and Application Insights resource name. The agent
> cannot retrieve telemetry data until this step is complete.

## How to Use These Prompts

- Copy any prompt below and use it directly in a conversation with the Monitoring Agent
- You can interact with the agent through:
  - **Microsoft Teams** via the agent added under *Built for your org*
  - **M365 Copilot** via the agent panel in the right navigation
- Prompts are flexible. The agent uses AI to match your intent, so you don't need to
  use the exact wording listed here
- You can refine results by specifying parameters like time range, batch job ID, or entity name

# Writing Your Own Queries

The Monitoring Agent is entirely **data-driven**. Every capability you see above is a row
in the **MCSA Agent Queries** Dataverse table. Adding a new monitoring capability is as
simple as inserting a new row — no code changes, no redeployment, no republishing.

This section explains the query schema so you can extend the agent to cover any telemetry
scenario across any platform that exports to Application Insights.

## Query Schema

| Column | Type | Required | Description |
|---|---|---|---|
| **User Prompt** | Text | Yes | The canonical name shown in configuration (e.g., `Batch Exceptions`) |
| **Description** | Text | Yes | Natural language description of what the query returns. The agent uses this to understand the query's purpose |
| **Prompt Variations** | Text | Yes | Alternative phrasings users might type, separated by semicolons or line breaks. The more variations, the better the agent matches intent |
| **Query** | Text | Yes | The KQL query to execute against Application Insights |
| **Platform** | Text | Yes | Platform identifier (e.g., `D365FO`, `Dataverse`, `PowerAutomate`) |
| **Tags** | Text | No | Comma-separated tags for categorization (e.g., `Batch, Errors, F&O`) |
| **Active** | Boolean | Yes | `True` to enable, `False` to disable without deleting |
| **Display Format** | Text | No | JSON object controlling how columns are rendered in the response (see below) |
| **Chart Type** | Text | No | `Bar Chart`, `Pie Chart`, `Column chart`, or `Time Chart` |
| **Render Chart** | Boolean | No | Set to `True` if the query should return a visual chart |
| **Daily Briefing** | Boolean | No | Set to `True` to include this query in the automated daily briefing report |
| **Anomaly Detection** | Boolean | No | Set to `True` if the query uses time-series anomaly detection |

## KQL Parameters

Queries support dynamic parameters using the `{ParameterName:'default'}` syntax.
The agent extracts parameter values from the user's message and injects them into the query at runtime.

| Parameter | Default | Description |
|---|---|---|
| `{TimeRange:'1d'}` | 1 day | Time window for the query (e.g., `15m`, `3h`, `7d`, `30d`) |
| `{TopN:'10'}` | 10 | Maximum number of results to return |
| `{BatchJobId:'""'}` | empty | Filter results by a specific batch job ID |
| `{TimeBucket:'10m'}` | 10 min | Time bucket granularity for aggregations |
| `{EntityName:'""'}` | empty | Filter by a specific entity name (DMF queries) |
| `{ExecutionId:'""'}` | empty | Filter by a specific execution ID |
| `{ActivityId}` | — | Trace a specific Activity ID across all telemetry tables |

You can define custom parameters using the same syntax. The agent will prompt
the user if a required parameter is not provided.

## Display Format

The `Display Format` column accepts a JSON object that controls how query result columns
are presented. Each key matches a column name from the KQL output.

```json
{
  "timestamp":    { "alias": "Time" },
  "outerMessage": { "alias": "Message", "max": 40 },
  "BatchJobId":   { "alias": "JobId", "max": 11 },
  "CallStack":    { "hide": true },
  "Count":        { "alias": "Count", "align": "right" }
}
```

| Property | Values | Description |
|---|---|---|
| `alias` | any string | Display name for the column header |
| `max` | number | Maximum character width before truncation |
| `hide` | `true` | Column is available for analysis but not shown in the table |
| `align` | `left`, `right`, `center` | Column alignment |
| `icon` | `threshold`, `arrow` | Visual indicators for values |
| `warn` / `crit` | number | Threshold values when `icon` is `threshold` |
| `format` | `percent` | Format values as percentages when `icon` is `arrow` |

## Query Example

Here is a complete example of a query that monitors failed cloud flow runs in Power Automate.
This example uses the official telemetry schema documented by Microsoft.

**User Prompt**: `Failed Cloud Flow Runs`

**Description**: `Retrieve recent cloud flow runs that ended in failure.`

**Prompt Variations**:
```
Show me failed flow runs
Which cloud flows failed recently?
Display Power Automate failures
Show flow errors
List failed flows
```

**Query**:
```kql
// === Failed Cloud Flow Runs ===
// ====== PARAMETERS ======
let _timeRange = {TimeRange:'1d'};
let _topN = toint({TopN:'10'});
// ====== BASE EVENTS ======
requests
| where timestamp > ago(totimespan(_timeRange))
| where customDimensions['resourceProvider'] == 'Cloud Flow'
| where customDimensions['signalCategory'] == 'Cloud flow runs'
| where success == false
| order by timestamp desc
| take _topN
```

**Display Format**:
```json
{
  "timestamp": { "alias": "Time" },
  "name":      { "alias": "Flow", "max": 30 },
  "success":   { "alias": "Status" }
}
```

**Platform**: `PowerAutomate` · **Active**: `True` · **Tags**: `Flows, Errors, PowerAutomate`

> 📖 **Reference**: [Set up Application Insights with Power Automate](https://learn.microsoft.com/power-platform/admin/app-insights-cloud-flow)
> — Cloud flow runs map to the `requests` table; triggers and actions map to `dependencies`.

### Configuration Options

The Configuration Center provides an interactive Adaptive Card with four management areas:

- **Manage Daily Summary Recipients**: View, add, or remove email recipients for the daily briefing report
- **Manage Alert Recipients**: View, add, or remove recipients for anomaly detection alerts

## Tips for Getting Better Responses

| Tip | Example |
|---|---|
| **Specify a time range** | Instead of *"Show batch errors"*, try *"Show batch errors for the last 3 hours"* |
| **Filter by ID** | Add a batch job ID: *"Batch execution history for Batch Id 48291"* |

| **Ask what's available** | If unsure, ask: *"What can I ask you?"* or *"Show me available queries"* |

## Out-of-Scope Topics

The Monitoring Agent is designed to analyze telemetry data from Application Insights.
If a question falls outside the configured query library or the agent's scope, it will
let you know and suggest alternatives.

Topics that are out of scope include:

- Direct changes to D365 environment configurations
- Real-time transactional data from Dynamics 365 (the agent reads telemetry, not live business data)
- Questions unrelated to system health, performance, or telemetry
- Creating or modifying KQL queries through the chat interface
