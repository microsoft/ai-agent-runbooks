# Step-by-Step Runbook — F&O User Onboarding Agent

> **Scenario**: Automated User Provisioning for Dynamics 365 Finance & Operations
> **Platform**: Microsoft Copilot Studio · Power Automate · Microsoft Graph API · Dataverse · D365 F&O
> **Target Readers**: Delivery engineer, Power Platform admin, IT admin
> **Last Updated**: April 2026

---

## Overview

This runbook walks you through the complete setup and deployment of the **F&O User Onboarding Agent**.
By the end of this guide, the agent will be able to:

- Accept a structured onboarding request from an HR or IT admin via Microsoft Teams / M365 Copilot
- Validate or automatically create the user in Microsoft Entra ID using the Graph API
- Route the request for manager/admin approval via a Teams Adaptive Card (Mode 1 only)
- Provision the user in the correct D365 F&O company with the appropriate security roles
- Return a success or failure notification to the requestor

## Phase 0: F&O X++ Solution Deployment & Virtual Entity Setup

> ⚠️ **This phase must be completed first — before importing the Power Platform solution.**
> The F&O X++ model (`CSAOnboardingAgent`) adds the custom entities, data entities, and
> service operations that the agent's Power Automate flows depend on. Without it deployed
> to the F&O environment, provisioning calls will fail at runtime.

---

### Step 0-1. Import the X++ Solution into Visual Studio and Check into Azure DevOps

The F&O solution is packaged as an **`.axpp` file** (`CSAOnboardingAgent.axpp`).
This file must be imported into Visual Studio, compiled, and committed to your Azure DevOps
repository so it can be built and deployed through a standard pipeline.

1. Open **Visual Studio** with the **Dynamics 365 Finance and Operations** extension installed
2. Go to **Extensions** → **Dynamics 365** → **Import Project**
3. Browse to and select: `CSAOnboardingAgent.axpp`
4. Choose the target model (create a new model or add to an existing extension model)
5. Click **OK** — Visual Studio imports and creates the project from the axpp

   > 💡 Recommended model naming: `CSAOnboardingAgent` with publisher prefix matching your org

6. In **Solution Explorer**, right-click the project → **Add to Source Control**
7. In **Team Explorer** → push the project to your Azure DevOps repository branch

   | Git Action | Value |
   |---|---|
   | Repository | Your D365 F&O Azure DevOps repo |
   | Branch | `feature/fo-onboarding-agent` (or your naming convention) |
   | Commit message | `feat: Add CSAOnboardingAgent X++ model` |

8. Raise a Pull Request into `main` / `release` branch following your standard code review process

---

### Step 0-2. Build and Create a Deployable Package (One Version)

Once the code is merged, build a **Unified Deployable Package** that can be applied to any
UDE or cloud-hosted F&O environment — including environments accessible via LCS.

#### Option A — Azure DevOps Build Pipeline (Recommended)

1. In **Azure DevOps**, navigate to **Pipelines** → your existing D365 F&O build pipeline
2. Trigger a new build against the branch containing the `CSAOnboardingAgent` model
3. The pipeline compiles the X++ code and produces a **Deployable Package** (`.zip`) as a build artifact
4. Download the deployable package artifact from the pipeline run

   > 💡 The pipeline must include the `CSAOnboardingAgent` model in its build scope.
   > If using a standard Microsoft build pipeline template, ensure the model is listed
   > under the **Package** step in the pipeline YAML.

#### Option B — Local Build (UDE / Developer Machine)

1. In **Visual Studio**, open the `CSAOnboardingAgent` project
2. Go to **Dynamics 365** → **Build models** → select `CSAOnboardingAgent`
3. After successful compilation, go to **Dynamics 365** → **Create deployable package**
4. Select the output folder and click **Create**
5. The tool generates a `.zip` deployable package ready for LCS or direct UDE deployment

   | Package Details | Value |
   |---|---|
   | Package file | `CSAOnboardingAgent_<version>.zip` |
   | Package type | Software deployable package (X++ hotfix) |
   | Target environments | UDE, Cloud-hosted, Sandbox (Tier 2+) |

---

### Step 0-3. Upload and Deploy the Package

#### Deploy via LCS (Sandbox / Production environments)

1. Sign in to **Lifecycle Services** → https://lcs.dynamics.com
2. Open your project → go to **Asset library** → **Software deployable package**
3. Click **+ Upload** and upload the `CSAOnboardingAgent_<version>.zip` file
4. Once uploaded, click **Confirm** to validate the package
5. Go to your target environment → **Maintain** → **Apply updates**
6. Select the `CSAOnboardingAgent` package from the asset library → click **Apply**
7. Wait for the deployment to complete and the environment to restart

   > ⚠️ Sandbox (Tier 2+) deployments require a scheduled maintenance window.
   > Production deployments require LCS change management approval.

#### Deploy directly to a UDE (Unified Developer Environment)

For UDE environments, you can sync directly from Azure DevOps instead of using LCS:

1. On the UDE virtual machine, open a **Developer command prompt for Visual Studio**
2. Run a sync from Azure DevOps to pull the latest model:
   ```
   xppcmd.exe /command:Synchronize
   ```
3. Alternatively, in Visual Studio → **Team Explorer** → pull the latest changes → rebuild
4. After the build, run a full database sync:
   - Go to **Dynamics 365** → **Synchronize database** (or run `SyncEngine` from the command line)

### Step 0-6. Enable F&O Virtual Entities in Power Platform

Virtual entities allow the agent's Power Automate flows and Dataverse to read and write
F&O data (users, security roles, companies) without data replication. Each entity must
be explicitly activated before it can be queried.

#### 0-6a. Confirm the Virtual Entity Solution is Installed

4. If the solution is not present, install it from **AppSource**:

#### 0-6b. Activate Individual Virtual Entities

Each F&O entity used by the agent must be individually activated (made **Visible**) in Dataverse.

1. In the **Power Apps Maker Portal**, go to **Tables** → click **All** (to show all tables, not just default)
2. In the search box, search for each of the following entities required by the onboarding agent:

   | Virtual Entity | F&O Table / Purpose |
   |---|---|
   | `LicensingRoleMappingEntity` | Maps licensing roles to D365 F&O security roles — required for automated role assignment during onboarding |

3. For each entity, click to open it → look for the **Visible** property in the table settings
4. Set **Visible** = **Yes** → click **Save**

   > **Alternative path from F&O** (if the Power Apps method is unavailable):
   > 1. In D365 F&O, go to **System Administration** → **Setup** → **Virtual entities**
   > 2. Search for the entity by name
   > 3. Set **Enabled** = **Yes** → click **Save**
   > 4. The entity will become visible in Dataverse within a few minutes

   > ⚠️ Only activate the entities your flows actually query. Activating all entities
   > unnecessarily increases the surface area for API access and can degrade performance.

## Phase 1: Azure App Registration (Graph API Access)

This phase creates the Azure App Registration that allows the agent's Power Automate
flows to authenticate against the Microsoft Graph API for Entra ID operations.

### Step 1-1. Create the App Registration

1. Sign in to the **Azure Portal** → https://portal.azure.com
2. Navigate to **Microsoft Entra ID** → **App registrations** → **+ New registration**
3. Fill in the following values:

   | Field | Value |
   |---|---|
   | Name | `FOUserOnboardingAgent` (or your preferred name) |
   | Supported account types | Accounts in this organizational directory only (Single tenant) |
   | Redirect URI | Leave blank |

4. Click **Register**
5. On the app's overview page, copy and save the following values — you will need them later:
   - **Application (client) ID**
   - **Directory (tenant) ID**

---

### Step 1-2. Assign API Permissions

The permissions required depend on the operational mode you are deploying:

> 💡 Start with the **minimum required permissions** for your environment's mode.
> You can always add permissions later if you switch modes.

**For Mode 2 (Validate Only):**

1. In the app registration, go to **API permissions** → **+ Add a permission**
2. Select **Microsoft Graph** → **Application permissions**
3. Search and add: `User.Read.All`
4. Click **Add permissions**
5. Click **Grant admin consent for [your tenant]** → **Yes**

**For Mode 1 or Mode 3 (Auto-Create User):**

1. In the app registration, go to **API permissions** → **+ Add a permission**
2. Select **Microsoft Graph** → **Application permissions**
3. Search and add each of the following:
   - `User.ReadWrite.All`
   - `Directory.ReadWrite.All`
4. Click **Add permissions**
5. Click **Grant admin consent for [your tenant]** → **Yes**

> ⚠️ **Admin consent is required.** Without it, the API calls will return 403 Forbidden.
> The user performing this step must be a **Global Administrator** or **Application Administrator**.

---

### Step 1-3. Create a Client Secret

1. In the app registration, go to **Certificates & secrets** → **+ New client secret**
2. Fill in:

   | Field | Value |
   |---|---|
   | Description | `FOUserOnboardingAgent-Secret` |
   | Expires | 12 months (or per your org's policy) |

3. Click **Add**
4. **Copy the secret value immediately** — it will not be shown again after you leave this page
5. Store the secret securely; you will add it to Dataverse in Phase 2

### Step 3-2. Configure the OnboardAgentConfig Table

The `OnboardAgentConfig` table in Dataverse is the **single configuration source** for the
agent. You must populate it before the agent can function.

1. In the **Power Apps Maker Portal**, go to **Dataverse** → **Tables**
2. Find and open the **OnboardAgentConfig** table
3. Click **Edit** to open the table data editor
4. Create a new row with the following values:

   | Column | Value | Notes |
   |---|---|---|
   | `CreateADUser` | `1` or `0` | `1` = auto-create Entra ID user; `0` = validate only |
   | `ApprovalRequired` | `Yes` or `No` | Only relevant when `CreateADUser = 1` |
   | `TeamName` | **`F&O User Provisioning`** | Must match the Teams team name exactly (case-sensitive) — see Step 2-4a |
   | `ChannelName` | **`FNOUserApprove`** | Must match the channel name exactly (case-sensitive) — see Step 2-4b |
   | `GraphClientID` | Application (client) ID from Step 0-1 | From Azure App Registration |
   | `GraphClientSecret` | Client secret value from Step 0-3 | Treat as sensitive — do not share |
   | `GraphTenantID` | Directory (tenant) ID from Step 0-1 | From Azure App Registration |

   > 📋 **Teams reference values for this deployment** (from Step 2-4):
   >
   > | Property | Value |
   > |---|---|
   > | Team Name | `F&O User Provisioning` |
   > | Channel Name | `FNOUserApprove` |
   > | Group ID | `28b69a3d-b7af-4f5b-be42-a53bd17dd6c4` |
   > | Channel ID | `19%3A0oFu9NU3FTF6Dh1Y5CGCCOJx2yYl1XfCe5HAl-Gajko1%40thread.tacv2` |
   > | Tenant ID | `1557f771-4c8e-4dbd-8b80-dd00a88e833e` |
   >
   > If your Power Automate flow uses **Group ID** and **Channel ID** directly (instead of
   > display names), use the values above in the corresponding flow action fields.

5. Click **Save**

> 🔒 **Security note**: The `OnboardAgentConfig` table contains Graph API credentials.

### Step 4-3. Review Agent Topics and Prompts

The agent exposes **three direct conversational prompts only**. All other interactions
(collecting user details, role selection, approval responses) are handled through
**Adaptive Cards** returned by the flows — not by additional topics.

1. Go to **Topics** in the left navigation
2. Confirm the following three topics exist and are enabled:

   | Topic / Prompt | Trigger Phrase | What It Does |
   |---|---|---|
   | **Onboard User** | `Onboard user` | Starts the guided onboarding flow — collects new hire details via Adaptive Card |
   | **Security Posture** | `Security posture of the user` | Returns the current security role assignments for a specified D365 F&O user |
   | **Offload User** | `Offload user` | Triggers the offboarding/deprovisioning workflow for an existing user |

3. Disable or remove any other topics not listed above — the agent is designed to handle
   only these three entry points; all remaining interactions are driven by Adaptive Cards
4. Open each topic and confirm it correctly calls the corresponding Power Automate flow via the action step

---

### Step 4-4. Test the Agent in Copilot Studio

1. Click **Test** in the top-right corner of Copilot Studio
2. Start a new conversation with the message: **"I need to onboard a new F&O user"**

## Troubleshooting Guide

### Issue: Teams Adaptive Card not appearing (Mode 1)

| Symptom | Likely Cause | Resolution |
|---|---|---|
| Card never appears in Teams channel | `TeamName` or `ChannelName` in `OnboardAgentConfig` is incorrect | Confirm values are exactly `F&O User Provisioning` and `FNOUserApprove` (case-sensitive — no trailing spaces) |
| Card never appears despite correct names | Flow uses Group ID / Channel ID and these are stale | Verify Group ID = `28b69a3d-b7af-4f5b-be42-a53bd17dd6c4` and Channel ID = `19%3A0oFu9NU3FTF6Dh1Y5CGCCOJx2yYl1XfCe5HAl-Gajko1%40thread.tacv2` in the flow action |
| Card appears but approval button fails | Teams connection in the approval flow is using wrong account | Re-authenticate the Teams connector in Power Automate with the service account that has access to `FNOUserApprove` |
| Flow times out waiting for approval | Approval card was not acted on within the flow timeout window | Increase the approval wait timeout in the Teams Approval Flow, or notify the `FNOUserApprove` channel approvers directly |

---

### Issue: Graph API returning 403 Forbidden

| Symptom | Likely Cause | Resolution |
|---|---|---|
| `403 Forbidden` on user lookup | Admin consent not granted for API permissions | Go to Azure Portal → App Registration → API permissions → Grant admin consent |
| `403 Forbidden` on user creation | `User.ReadWrite.All` or `Directory.ReadWrite.All` not granted | Add permissions and re-grant admin consent (Step 0-2) |

---

### Issue: D365 F&O provisioning fails

| Symptom | Likely Cause | Resolution |
|---|---|---|
| User not appearing in D365 F&O after flow | D365 F&O connection is using insufficient permissions | Confirm the service account used in the D365 F&O flow connection has **System Administrator** role in the target company |
| Security role not assigned | Role name does not match exactly | Verify the security role name in the onboarding form matches the internal D365 F&O role name (case-sensitive) |
| Company code not recognized | Legal entity code is incorrect | Confirm the company code (e.g., `USMF`, `GBSI`) matches the D365 F&O company's exact identifier |

---

### Issue: Agent not responding or showing error in chat

| Symptom | Likely Cause | Resolution |
|---|---|---|
| Agent gives generic error response | Flow connection references not configured | Open each flow in Power Automate → verify all connections are bound → turn flow off and on again |
| Agent starts but form does not complete | Topic action step is not correctly bound to the flow | In Copilot Studio → Topics → Onboarding Form → verify the action step calls the correct flow |
| Agent not available in Teams | Agent not published or Teams channel not configured | Republish in Copilot Studio → re-add to Teams channel |
